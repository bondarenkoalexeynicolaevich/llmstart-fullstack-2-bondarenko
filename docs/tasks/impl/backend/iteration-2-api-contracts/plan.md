# Итерация 2 (backend): контракты API

## Цель и ценность

Зафиксировать канонический REST-контракт v1 для двух MVP-сценариев: **вопрос ассистенту** (один запрос — ответ с учётом истории и `Flow.system_prompt`) и **фиксация сдачи ДЗ** (`Submission`). Снять неоднозначность идентификации ботом и ошибок до реализации в итерации 5.

**Файл OpenAPI 3.x:** [`docs/api/backend-v1.openapi.yaml`](../../../../api/backend-v1.openapi.yaml).

Markdown-детализация по сценариям — в задачах ниже; при расхождении текста с OpenAPI приоритет у **YAML**.

## Аутентификация bot → backend (MVP)

| Аспект | Решение |
|--------|---------|
| Механизм | Заголовок `Authorization: Bearer <token>` |
| Секрет | Одно значение в `.env` у backend и у бота: `INTERNAL_API_TOKEN` (не логировать, не отдавать клиентам) |
| Нет заголовка / неверный токен | `401`, код ошибки `unauthorized` |
| Участник не может выполнить операцию по роли потока (например ожидается студент) | `403`, код `forbidden` |

Детали размещения переменных — в [`docs/integrations.md`](../../../../integrations.md) и [`.env.example`](../../../../../.env.example).

## Идентификация вызывающей стороны и участника

Бот знает **целочисленный** `telegram_user_id` (Telegram user id). В теле запросов MVP **всегда** передаются:

| Поле | Тип | Назначение |
|------|-----|------------|
| `flow_id` | UUID | Поток; без него при нескольких потоках у пользователя контракт неоднозначен |
| `telegram_user_id` | integer | Резолв `User` (по `telegram_id`) → `Participant` для данного `flow_id` на стороне backend |

`participant_id` в теле **не** передаётся: выводится сервером. Если пользователя нет в системе или нет `Participant` для пары (user, flow) — `404` с уточняющим `code` (см. матрицу и OpenAPI).

## Таблица endpoint’ов (v1)

| Метод и путь | Назначение | Успех |
|--------------|------------|--------|
| `POST /v1/dialog-messages` | Пройти историю, сохранить сообщение пользователя, сгенерировать ответ ассистента (логика LLM — итерация 5), сохранить ответ | `200` + тело с текстом ответа и идентификаторами сообщений |
| `POST /v1/submissions` | Создать `Submission` для пары (participant, assignment) | `201` + созданный ресурс |

Чтение `GET /v1/submissions/...` в контракт MVP **не входит** (бот может обойтись ответом `201`); при появлении сценария — отдельная итерация контракта.

## Матрица ошибок (обобщённо)

Все ошибки — объект верхнего уровня `error` с полями `code` (стабильный идентификатор для клиента), `message` (человекочитаемо, не для ветвления), опционально `details` (например поля валидации).

| HTTP | `code` (примеры) | Когда |
|------|-------------------|--------|
| `400` | `validation_error` | Невалидное тело / пустой `content` у диалога |
| `401` | `unauthorized` | Нет или неверный Bearer |
| `403` | `forbidden` | Роль в потоке не позволяет операцию |
| `404` | `flow_not_found`, `participant_not_found`, `assignment_not_found` | Нет сущности или нет участия пользователя в потоке |
| `409` | `submission_already_exists` | Уже есть `Submission` на эту пару participant + assignment |
