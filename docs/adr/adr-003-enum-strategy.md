# ADR-003: Перечисления домена в PostgreSQL (native ENUM)

| Атрибут | Значение |
|---|---|
| Статус | Принято |
| Дата | 2026-04-19 |
| Контекст | Ограничение допустимых значений для ролей и статусов в схеме БД; стек ORM из [ADR-002](adr-002-backend-http-orm.md) |

---

## Контекст

В модели есть небольшие закрытые множества значений: роль участника, роль сообщения диалога, статус сдачи. Их нужно согласованно отразить в DDL и в SQLAlchemy, не размножая «магические строки» без контроля со стороны БД.

Уже созданные в миграциях типы PostgreSQL:

- `member_role` — `student`, `teacher`
- `dialog_message_role` — `user`, `assistant`, `system`
- `submission_status` — `submitted`, `reviewed`, `approved`

В ORM используются `enum.Enum` в Python и `sqlalchemy.Enum(..., native_enum=True)` с теми же именами типов.

---

## Рассмотренные варианты

### 1. Native PostgreSQL `ENUM` (`CREATE TYPE … AS ENUM`)

**Плюсы:** явный тип в БД, совпадение с `native_enum=True` в SQLAlchemy, проверка на уровне СУБД.

**Минусы:** изменение набора значений требует миграций (`ALTER TYPE … ADD VALUE` и т.д.); при переиспользовании одного типа в нескольких таблицах в Alembic нужно аккуратно задавать `create_type` / `values_callable` (см. миграцию `001_initial`).

### 2. `TEXT` + `CHECK` (или без CHECK, только приложение)

**Плюсы:** проще добавлять новые значения без особенностей PG ENUM.

**Минусы:** дублирование ограничений между приложением и БД; для проекта уже выбран и применён native ENUM — откат потребовал бы новых миграций.

---

## Решение

**Оставляем native PostgreSQL ENUM** и в миграциях (`sqlalchemy.dialects.postgresql.ENUM`), и в моделях (`SAEnum(..., name="…", native_enum=True)`), в соответствии с уже существующими ревизиями.

**Правила в Alembic:**

- Первое использование типа: `postgresql.ENUM(..., name="…", create_type=True)`.
- Повторное использование того же имени типа в той же ревизии: отдельный объект с `create_type=False` и `values_callable`, совпадающим с порядком/набором значений (как `member_role_existing` в `001_initial`).
- В `downgrade()` после `drop_table` для таблиц, использующих тип, выполнять `DROP TYPE IF EXISTS имя_типа` в корректном порядке (зависимости от таблиц).

**Добавление нового значения в существующий тип:** через `ALTER TYPE … ADD VALUE` в новой миграции; учитывать ограничения PostgreSQL (в частности, до версии 12 новое значение нельзя было добавить внутри блока транзакции — в актуальных версиях PG правила смягчены, но при сомнениях сверяться с документацией целевой версии).

---

## Последствия

- Новые доменные перечисления по возможности оформлять так же: Python `enum.Enum` + `SAEnum(..., native_enum=True)` + миграция с `postgresql.ENUM` и явным `DROP TYPE` при откате.
- Code review миграций с enum: проверять `create_type`, порядок `DROP TYPE`, отсутствие дублирующего `CREATE TYPE` для одного имени.
- Альтернатива TEXT+CHECK для *новых* сущностей допустима только при явном обосновании (например частые изменения набора значений) и отдельном согласовании — по умолчанию следуем этому ADR.
