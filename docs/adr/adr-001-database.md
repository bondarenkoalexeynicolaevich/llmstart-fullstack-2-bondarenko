# ADR-001: Выбор СУБД

| Атрибут | Значение |
|---|---|
| Статус | Принято |
| Дата | 2026-03-31 |
| Контекст | Проектирование системы сопровождения учебного потока, MVP |

---

## Контекст

Системе нужно хранить структурированные взаимосвязанные данные: пользователей, потоки, структуру курсов, прогресс студентов, историю диалогов с AI-ассистентом.

Ключевые характеристики нагрузки на MVP:
- Десятки пользователей одновременно.
- Частые чтения (прогресс, история диалогов), умеренные записи (новые сообщения, submission'ы).
- Данные строго структурированы и взаимосвязаны.

В перспективе — потенциальная необходимость векторного поиска для RAG по материалам курса.

---

## Рассмотренные варианты

### 1. PostgreSQL

Реляционная СУБД с открытым исходным кодом.

**Плюсы:**
- Строгая схема — надёжная работа со связанными сущностями (User → Participant → Submission).
- Транзакции и целостность данных из коробки.
- Расширение `pgvector` позволяет добавить векторный поиск без смены СУБД.
- Отлично поддерживается в Python-экосистеме: `asyncpg`, `SQLAlchemy`, `Alembic`.
- Широкая документация, долгосрочная поддержка.

**Минусы:**
- Требует настройки схемы и миграций.
- Немного тяжелее для старта, чем SQLite.

---

### 2. SQLite

Встроенная файловая БД, не требует отдельного процесса.

**Плюсы:**
- Нулевая инфраструктура — один файл, работает сразу.
- Удобно для ранних прототипов и локальной разработки.

**Минусы:**
- Нет конкурентной записи — при нескольких одновременных пользователях возникают блокировки.
- Сложно мигрировать на более серьёзную СУБД позже.
- Нет поддержки `pgvector` или аналогичного расширения.
- Не подходит для production-среды.

---

### 3. MongoDB

Документо-ориентированная NoSQL БД.

**Плюсы:**
- Гибкая схема — удобно для слабоструктурированных данных.
- Хорошая горизонтальная масштабируемость.

**Минусы:**
- Данные системы хорошо структурированы — гибкость схемы не нужна и создаёт риски.
- Связи между сущностями сложнее поддерживать без JOIN.
- Для векторного поиска нужен Atlas (платный) или сторонний сервис.
- Дополнительная экосистема (ODM, драйверы) без явной пользы на данном этапе.

---

## Решение

**Выбран PostgreSQL.**

Реляционная модель данных системы (пользователи, потоки, прогресс) — прямое попадание в сильные стороны PostgreSQL. Расширение `pgvector` позволит добавить RAG-поиск по материалам курса без смены стека. Экосистема Python хорошо поддерживает PostgreSQL, а инфраструктурные издержки оправданы уже на MVP.

SQLite исключён из-за ограничений на конкурентную запись. MongoDB — из-за отсутствия реального преимущества при структурированной доменной модели.

---

## Последствия

- В `backend/` используется `asyncpg` или `SQLAlchemy[asyncio]` как ORM/драйвер.
- Миграции управляются через `Alembic`.
- В локальной разработке PostgreSQL запускается через Docker Compose.
- При необходимости RAG подключается `pgvector` как расширение к той же БД — без миграции на другую СУБД.
