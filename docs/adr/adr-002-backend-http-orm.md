# ADR-002: HTTP-слой, доступ к данным и миграции (backend)

| Атрибут | Значение |
|---|---|
| Статус | Принято |
| Дата | 2026-04-05 |
| Контекст | Серверное ядро на Python поверх PostgreSQL ([ADR-001](adr-001-database.md)) |

---

## Контекст

Нужно зафиксировать прикладной стек для `backend/`: как принимать HTTP-запросы от клиентов (бот, веб), как работать с БД асинхронно и как версионировать схему. ADR-001 выбрал СУБД; здесь — инструменты в коде.

---

## Рассмотренные варианты

### 1. FastAPI + SQLAlchemy 2.x (async) + asyncpg + Alembic + uvicorn

**Плюсы:**

- FastAPI даёт маршруты, валидацию и OpenAPI из коробки, хорошо стыкуется с Pydantic.
- SQLAlchemy 2.x с async и драйвером `asyncpg` — распространённый стек для PostgreSQL в asyncio.
- Alembic — стандарт миграций для SQLAlchemy, один явный источник DDL.
- uvicorn — обычный ASGI-сервер для FastAPI.

**Минусы:**

- Несколько зависимостей; команде нужно понимать async-сессии и миграции.

---

### 2. Starlette (или чистый ASGI) без FastAPI, ручная валидация / сторонние схемы

**Плюсы:**

- Минимальный каркас, меньше «магии».

**Минусы:**

- OpenAPI и удобные схемы запросов/ответов придётся собирать отдельно; для MVP с контрактами API это лишняя работа.

**Отклонено:** избыточная ручная работа без выигрыша в простоте для нашего кейса.

---

### 3. Синхронный SQLAlchemy + psycopg в thread pool

**Плюсы:**

- Привычная модель для части разработчиков.

**Минусы:**

- Смешение sync ORM с async FastAPI усложняет код и повышает риск блокировок event loop при неверном использовании.

**Отклонено:** предпочитаем единообразный async-путь для I/O к БД под asyncio.

---

## Решение

**Приняты:** **FastAPI**, **SQLAlchemy 2.x** в async-режиме с **asyncpg**, **Alembic** для миграций, **uvicorn** как ASGI-сервер.

Согласуется с [`docs/vision.md`](../vision.md) (Python 3.12+, uv, ruff, dotenv) и с последствиями [ADR-001](adr-001-database.md).

---

## Последствия

- Новый код доступа к данным в `backend/` пишется в терминах async SQLAlchemy и миграций Alembic.
- Локальный запуск приложения — через uvicorn (конкретные команды и Makefile — по мере появления кода, см. tasklist-backend, итерации 3 и 8).
- При смене HTTP-фреймворка или ORM — новый ADR; этот документ не переписывается задним числом.
